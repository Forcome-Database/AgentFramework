## Applies When
- 全部项目，无条件适用。

## Do Not Apply When
- 无。

## Output Target
`## 反复提醒沉淀`

## Rule
- 如果开发过程中总是遇到某个问题，或者用户反复提醒同一个注意事项，先执行回流三判定：一，去掉项目名和具体路径后这条规则仍然成立吗？二，在另一个同类项目里 AI 会犯同样的错吗？三，这条规则不依赖本项目独有的架构决策、业务概念或第三方服务吗？
- 三条全部为「是」：把它抽象成规则块，补齐五字段，写入框架 git 工作副本的对应分类目录，并在回复中明确告知「已回流规则块」及其 id。
- 不要把回流写进插件缓存目录 —— 因为 `~/.claude/plugins/cache/` 下的目录在插件更新时被整个替换，写进去的规则会无声消失。
- 任一为「否」：只写入本文件对应章节。找不到合适章节时写入 `## 项目注意事项`。
- 补充时写成明确、可执行的规则。禁令写成「不要做 X —— 因为 Y（证据 Z）」。
- 不要静默地把规则回流到框架素材库 —— 因为回流会影响未来所有项目，不告知等同于污染。
- 不要把踩坑记录当作规则沉淀 —— 因为它们是事实而非约束，应写入 `docs/pitfalls.md`。

## Verification
- 自查：用户本轮是否重复了此前提过的要求？若是，是否已执行回流三判定并落盘？
- 自查：新写入的规则是否带有可执行的验证方式？
