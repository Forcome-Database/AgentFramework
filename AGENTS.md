# AGENTS.md

本文档用于约束本框架仓库中的 AI / 自动化开发行为。开发时优先遵循本文件，其次遵循用户当前消息。

本仓库是一个规则素材库与生成器，产出物是给其它项目使用的 `AGENTS.md`。因此本仓库对自身的约束必须至少和它施加给别人的一样严格。

## 基本原则

- 先读现有规则块，再新增规则块，优先沿用已有的写法与颗粒度。
- 不要臆测某条规则的来源 —— 因为无据可查的规则会被后来者当作噪音删除，每条规则必须带证据。
- 不要改无关文件，不要顺手重构 —— 因为它们会淹没本次改动的真实意图。
- 不要回滚或覆盖工作区中已有的用户改动 —— 因为那是用户未提交的工作。

## 规则块约束

- 新增规则块前，先检查是否已有规则块具有相同的 `Applies When`、`Do Not Apply When` 与 `Output Target`。三者相同则合并进该块，不新建文件。
- 五个字段必须齐全。不要提交缺 `Verification` 的规则块 —— 因为无法判定的规则等于没有规则。
- `legacy/` 分类下的规则块，`Do Not Apply When` 至少两条 —— 因为腐烂与健康常常同形（空目录既可能是废弃抽象层，也可能是正在重建的活目录），误杀的代价远大于漏报，一条排除条件挡不住同形误判。
- 标为 `可逆性：自动` 的规则块必须声明 `作用域`，且每项落在文档白名单内 —— 因为自动档会让 AI 直接动手，不划定写的边界就等于允许它改代码。
- 禁令必须写成三段式：不要做 X —— 因为 Y（证据 Z）。
- 不要使用「合理地」「适当地」「根据情况」这类措辞 —— 因为它们无法被判定。
- 规则块的 `id` 只增不改。已发布的 `id` 被其它文件交叉引用，改名会造成断链。
- 回流的规则块写进 git 工作副本，不要写进 `~/.claude/plugins/cache/` —— 因为插件缓存在插件更新时被整个替换，写进去的规则会无声消失。
- 每次改动规则库后必须运行 `node scripts/validate-rules.mjs`，返回 0 方可提交。
- 每次改动 README、两份手册、任一 plugin.json 或规则块后，必须运行 `node scripts/check-consistency.mjs`，返回 0 方可提交 —— 因为同一个事实散在多处（规则块数量在 5 个文件里，`legacy/` 档位在 4 个文件里），改一处忘另一处就是双文件分叉，而这在本框架的开发中发生过十次以上（见 `docs/pitfalls.md`）。

## 只读文件

- `reference/example-AGENTS.md` 是一份合成样例，描述一个不存在的项目，仅用于风格校准。
- 不要往 `reference/example-AGENTS.md` 里塞真实项目的内容 —— 因为它是样例污染门的禁词来源，真实内容会导致误判。
- 不要把 `reference/example-AGENTS.md` 中的任何条目复制进规则块 —— 因为它描述的项目并不存在，其中的路径与技术栈全是虚构的。

## INIT.md 约束

- `INIT.md` 的每一步必须可判定，须写明进入条件与退出条件。
- 不要在 `INIT.md` 中要求 AI「合理判断」 —— 因为生成器本身就是给 AI 读的文档，模糊指令会被自由发挥。

## 文档规范

- `AGENTS.md` 是唯一事实源。`CLAUDE.md` 全文仅一行 `@AGENTS.md`。
- 不要维护两份都含技术细节的约束文件 —— 因为它们必然分叉，而 agent 会捡到过期的那份。
- 踩坑记录写入 `docs/pitfalls.md`，不要追加进本文件。
- 本文件超过 300 行时停止追加，先按主题拆分到 `docs/` 下。
- 规则的证据汇总写入 `docs/evidence.md`。

## 项目注意事项

- 脚本零依赖，仅允许 `node:crypto`、`node:fs`、`node:path`、`node:url`、`node:test`、`node:assert/strict` 与全局 `fetch`。
- 判断脚本是否被直接调用时，必须用 `node:url` 的 `fileURLToPath` 解析 `import.meta.url` —— 因为 `new URL(...).pathname` 返回 percent-encoded 路径，仓库路径含非 ASCII 字符时比较必然失败且静默退出 0。
- 占位符检查绝不扫描 `reference/`、`INIT.md` 与 `REFACTOR.md` —— 因为这些文件按定义含被禁词字面量，扫描必然自匹配。
- 验收在真实项目的副本上进行。不要向被验收的仓库写入文件 —— 因为验收不得成为一次未经请求的改动。
