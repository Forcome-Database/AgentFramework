# 规则从哪来

本框架的规则不是从单个项目推断的，也不是从「最佳实践」文章里抄的。

素材来自对 17 份真实的 agent 约束文件（`AGENTS.md` / `CLAUDE.md`）和 8 个后端项目**实际代码结构**的调研。调研区分了自研项目与第三方 fork——第三方项目的约定反映的是上游作者的习惯，不是使用者的，因此不进入素材库，只借鉴其结构与写作范式。

为保护未公开的仓库，下文隐去项目名，但**数字与现象原样保留**。证据的力量在数字，不在名字：知道「8 个项目里只有 1 个用仓储层」就够了，知道那个项目叫什么并不增加任何说服力。

## 被推翻的四个假设

调研始于一个前提：把某个运转良好的项目的 `AGENTS.md` 通用化。结果推翻了其中四条。

### 一、统一响应封装不是通用约定

那个样板项目的 Go 后端用 `{code, data, msg}` 封装所有响应。8 个后端项目的实际代码说：

| 做法 | 项目数 |
| --- | --- |
| 直接返回类型化的 Pydantic `response_model`，错误用原生 `HTTPException` | 5 |
| 使用响应封装 | 2（均为早期或工具型项目） |
| 无 HTTP 层 | 1 |

近两年的 FastAPI 项目已经放弃了 Flask 时代的手写封装。把 `{code, data, msg}` 写进通用框架，等于让新项目倒退回已被淘汰的写法。

**不写入框架。** 但作为禁令写入：「不要给成功响应套封装 —— 因为类型化的 `response_model` 能被 OpenAPI 与客户端直接消费，而封装会抹掉类型。」

### 二、`crud/` 仓储层不是通用约定

8 个项目中仅 1 个使用泛型 CRUD 基类。其余在 service 层直接操作 session。

决定性的证据是另一个项目里一个**空的 `database/repositories/` 目录**——有人引入了仓储层，然后放弃了，只留下目录。

**不写入框架。** 作为禁令写入。

### 三、Alembic 不是通用约定

4 个使用关系型持久化的项目，给出 4 种不同的迁移方案：

- Alembic
- 手写 Flyway 风格的版本化 SQL 文件
- 自制迁移器类
- 启动时 `create_all`，不做迁移

**不写入框架。** 框架只要求：初始化时明确当前方案并写入 `## 项目注意事项`。

### 四、「不做迁移」与「不执行构建」不是可选项，是互斥对

同一个开发者的两个项目，在这两条上给出了**正面相反**的默认值：

> 项目 A：「项目尚未上线，不需要兼容旧数据……不写旧字段兼容、数据迁移兜底。」
> 项目 B：「改 DB 模型 → 配套迁移。测试库走 `create_all` 不读迁移，所以『测试过 ≠ 生产 schema 对』，**必须写迁移**。」

> 项目 A：「每次写完代码，不需要检查语法，不需要执行构建，用户会自己做。」
> 项目 B：「TDD 硬要求：先写用例（红）→ 实现到绿 → **提交前 `pytest -q` 全绿**。」

两条都正确。区别只在项目是否已上线、是否有 CI 门禁。

一条在特定前提下正确的规则，被无条件推广后会变成有害指令。项目上线后「不写迁移」会导致数据丢失；有 pre-commit 门禁时「不执行构建」会导致提交失败。

**因此它们组成两组互斥对**，`INIT.md` 阶段 3 强制二选一，零选或双选均判失败。

## 跨项目重复出现的规则

一条规则被多个互不相关的项目独立写下，是它具有通用性的最强证据。

### 外部知识查询优先级路由（3 个项目独立收敛）

三个项目各自总结出**同一条结构**，措辞不同：

> 「某浏览器自动化库的用法查对应 skill／钉钉文档用专用 MCP／其他依赖用 Context7／选择器问题用 Playwright MCP 验证」
>
> （另一项目几乎逐字相同）
>
> 「使用不熟悉的库或 API 前，按优先级：1. Context7（首选）2. 专用 MCP 3. WebFetch（兜底）」

第四个项目也有变体：某组件库的 API 查其官方的 `llms-full.txt`。

这是本次调研中通用性证据最强的规则，写入 `meta/external-knowledge-routing`。

### 单一权威来源（3 个项目出现）

选择器集中于选择器类、环境变量集中于 `Settings`、日志集中于 `setup_logging()`、类型集中于单一共享文件。另有项目明令「禁止在子应用中重复定义类型」「禁止创建空文件」。

### 不臆测，以一手事实为证据（2 次独立表述）

> 「先读现有代码，再动手修改。」
>
> 「不要猜，看日志、拉接口：网关异常直接 `curl` 真实接口验证，不臆测。」

同一元原则在「改代码前」与「排障时」的两个场景化表达。

## Python 后端的真实分层习惯

不看文档，只看实际代码结构。8 个项目的统计：

| 约定 | 命中 | 结论 |
| --- | --- | --- |
| 有数据库就分离持久化模型与传输模型 | **8/8 无一例外** | 写死为铁律 |
| `services/` 承载业务逻辑，路由函数保持薄 | 6/7 | 写入规则 |
| `lifespan` asynccontextmanager 做启停钩子 | 6/7 | 写入规则 |
| 配置集中于 `pydantic_settings.BaseSettings` | 4/7 | 写入规则，小脚本可例外 |
| `api/` vs `routers/` 命名 | 4 : 2 | **不写死**，两者皆可 |
| `crud/` / repository 层 | **1/8** | **不写入** |
| 统一响应封装 | 2/8，均为早期项目 | **不写入** |

规模最小的那个项目把所有端点直接写在 `main.py`，无 `routers/`、无 `services/`。这成了 `backend/python-fastapi-layering` 的 `Do Not Apply When`：端点少于 3 个时不适用。

## 观察到的反模式

详见 [`reference/anti-patterns.md`](../reference/anti-patterns.md)。摘要：

**双文件分叉。** 某财务自动化项目的一次修订把「多命中凭证按日期自动消歧」改为「一律交人工复核」，理由是自动选错即**静默错账**。该修订只写进了 `CLAUDE.md`，`AGENTS.md` 至今保留着旧规则。两份文件的同一行号给出正面矛盾的指令，任何读 `AGENTS.md` 的 agent 都会实现已被判定为危险的那个行为。

这直接导出了框架的 `meta/doc-governance`：`AGENTS.md` 是唯一事实源，`CLAUDE.md` 全文仅一行 `@AGENTS.md`。

**记忆文档无限膨胀。** 某项目的 `CLAUDE.md` 顶部写着「每次小版本迭代后须更新本文件」，却没有任何拆分、编号或归档规则。结果十天内从 10KB 涨到 146KB，51 条踩坑记录漂移成三种格式，顶部的状态摘要从未更新，已成误导信息。

只学会了「如何积累」，没学会「如何治理积累」。

## 借鉴自第三方开源项目的写作范式

第三方项目的**业务规则**不进素材库，但它们的**结构与写法**值得学：

- **LobeHub 官方**：`CLAUDE.md` 全文只有一行 `@AGENTS.md`，用 transclusion 从结构上杜绝漂移。禁令带成本理由：`NEVER run bun run test - takes ~10 minutes`。量化优于模糊：`After 2 failed fix attempts, stop and ask for help`。
- **Vercel `react-best-practices` skill**：每条规则固定为「量化影响 + 反例代码块 + 正例代码块 + `When NOT to use this pattern`」。

那个 `When NOT to use this pattern` 段落，和本框架在没看过它的情况下设计的 `Do Not Apply When` 字段，是同一个东西。两边独立收敛到同一结构，说明这不是巧合：**规则的适用边界必须和规则本身一起被写下来**，否则规则在边界外会变成有害指令。

## 单一样本的规则块

`backend/go-layering` 与 `backend/node-layering` 各自只有一个样本。它们的正文中明确声明：

> 本规则块基于单一样本，尚未获得跨项目验证。初始化时若目标项目结构不同，沿用目标项目既有结构，并通过双向沉淀回流修正本块。

`INIT.md` 阶段 7 会把这条声明原样报告给用户。规则的可信度必须和规则一起被交付。
