# REFACTOR.md

本文件是一份执行手册，读者是 AI。目标是扫描一个已有项目的存量文档与结构腐烂，整理能整理的，报告不能整理的。

四个阶段顺序执行。每个阶段有进入条件与退出条件。

本手册的每一步都必须可判定。若你发现某一步需要「合理判断」，说明手册有缺陷，停下来告诉用户。

本手册不生成 `AGENTS.md`。生成走 `INIT.md`。两份手册通过 `AGENTS.md` 这个产物解耦。

## 术语

- 框架根目录：**本文件（`REFACTOR.md`）所在的目录**。你是从某个绝对路径读到本文件的，那个目录就是框架根目录。不要去解 `${CLAUDE_PLUGIN_ROOT}` 之类的环境变量 —— 它是 Claude Code 专有的，Codex 与其它 agent 不认识它。下文所有 `reference/`、`scripts/` 路径均相对于框架根目录。
- 目标项目：要被整理的项目。下文凡未加「框架根目录」限定的相对路径，均相对于目标项目根目录。
- **决策日志**：目标项目的 `.agents/refactor-decisions.md`。**每一个「没有逐字保留原文」的决定都必须写进它**——去重、改写、消解矛盾、判定某章是踩坑日志。

  由各块 `Verification` 中的**零静默丢失核对**机器强制：原文的每一行，要么逐字存在于某个文件，要么出现在这份日志里，两者都不满足即阻断。

  **机器强制的不是「判断对不对」**（那需要另一个判断），**是「判断有没有被记录」**。一个没被记录的判断，等于没有判断：无从复核，也无从推翻。**该判断的就判断——你有一个大模型，用它。**

## 一个块写不写文件，看它有没有声明 `作用域`

```
Remediation 里有「作用域：...」   → 它会写这些文件，且只能写这些文件
Remediation 里没有「作用域」      → 它一个文件都不写，只报告
```

**这是一个事实，不是一个判断。**

曾经有过一个「可逆性」枚举（自动 / 报告），试图**提前静态地**判定一个动作安不安全。它在开发中翻改了四次——不是想不清楚，是这个问题在静态层面无解：**安不安全依赖具体项目，不依赖规则。**

现在：`legacy/orphan-abstraction` 与 `legacy/vendored-knowledge` 没有作用域，所以它们只报告。前者是因为 **git 根本不追踪空目录，删了就恢复不了**；后者是因为**目的地在仓库之外**（skill 或 Context7），框架管不到。这两个理由都是事实，不是对安全性的猜测。

## 阶段 0：前置

两个硬闸门，任一不过即终止本手册。

1. **`AGENTS.md` 必须存在于目标项目根目录。**不存在则告诉用户先运行 `init-agents`，然后停止。

   理由：阶段 1 要从 `AGENTS.md` 反查哪些规则块被选入。没有它就无从确定扫描集，只能全量扫描 —— 而全量扫描会在一个 Go 项目里报出 Python 的规则违规。那是噪音，不是发现。

2. **`git status --short` 必须为空。**不为空则列出脏文件，告诉用户先提交或暂存，然后停止。

   理由：工作区脏的时候动手整理，就无法区分「框架改的」和「用户没提交的」。这直接来自 `AGENTS.md` 的「不要回滚或覆盖工作区中已有的用户改动」。

**记录起始 commit：**`git rev-parse HEAD`，下文称 `<起始commit>`。

退出条件：两个闸门均通过，`<起始commit>` 已记录。

## 阶段 1：扫描

**本阶段禁止任何写操作。**

**扫描集** = `AGENTS.md` 中实际选入的规则块 ∩ 带 `Legacy Scan` 字段的块，并上**全部 `legacy/` 块**。

反查怎么做：读目标项目的 `AGENTS.md`，把框架根目录 `reference/rules/` 下各块 `Rule` 字段的每一条，拿去做**逐字全等的整行匹配**（`grep -Fqx -- "<条目原文>" AGENTS.md`）。有任何一条命中即判定该块被选入。

**当前规则库下这一部分的交集为空**——只有 `legacy/` 的 5 个块带 `Legacy Scan`，而它们已被无条件纳入。交集为空是正常结果，不要因此停下来。

对扫描集中每一块，逐条比对 `Applies When` 与 `Do Not Apply When`：

> **默认排除。**只有当 `Applies When` 每一条都在目标项目中找到证据，且 `Do Not Apply When` 每一条都不成立时，才判定为选入。有疑问时排除。

`Do Not Apply When` 在本手册中的语义与生成时不同：生成时它是「这条规则不适用于你的项目」，扫描时它是「**这个看起来像腐烂的东西其实是活的**」。**逐条跑它。它是唯一的防误杀机制。**

对选入的块，执行其 `Legacy Scan` 字段的命令。

退出条件：产出偏离清单，每行四列 —— 规则块 id、命中位置、会写哪些文件（有 `作用域` 则列出，无则写「只报告」）、`Do Not Apply When` 的逐条判定结果。零命中的块也要列出，标注「无命中」。

## 阶段 2：呈现

把完整的偏离清单给用户看。**动手之前，用户必须知道要动什么。**

不设批准闸门，呈现完即进入阶段 3。用户调用本 skill 这个动作本身就是授权；在一个有几百处违规的老项目上逐批征求同意，会变成几十轮问答，实际使用中用户会开始无脑点「是」——那时闸门就成了摆设，反而比没有更危险，因为它制造了「已经审过」的错觉。

用户看到清单后若不认可，直接中断即可。

退出条件：用户已看到完整清单。

## 阶段 3：执行

**只执行声明了 `作用域` 的块。**没有作用域的块不动任何文件，它们的命中留到阶段 4 报告。

### 顺序

**会往 `docs/` 新建文件的块先跑**（看 `作用域` 是否含 `docs/`）。

当前顺序：`legacy/memory-bloat` → `legacy/doc-index-rot` → `legacy/doc-fork`。

**顺序不是可有可无的。**`doc-fork` 要逐条判断 `CLAUDE.md` 的内容：哪条与 `AGENTS.md` 重复、哪条该进哪一节、哪两条矛盾。若它先跑，就得在一份混着架构图、配置表与踩坑日志的 682 行文档里做这些判断——**判断质量必然差**，而架构图和配置表根本不该进约束文件。

`legacy/doc-fork` 的 `Do Not Apply When` 里写死了兜底：`CLAUDE.md` 超过 300 行时它不适用。万一顺序被打乱，它会自己拒绝执行。

### 每个块，六步

1. **动手前 `git status --short` 必须为空。**非空即中止该块，转入报告，继续下一块。

   阶段 0 只在整轮开始时跑一次。执行到第二个块时，第一个块已经改过文件。

2. **重新判定本块的 `Applies When` 与 `Do Not Apply When`。**前一个块可能改变了它的适用性 —— `memory-bloat` 往 `docs/` 新建文件后，`doc-index-rot` 才真正适用。

   逐块重判，**不需要收敛循环**：顺序已经保证了「会新建文件的块」排在前面。

3. **执行 `Remediation` 的动作。**每一个「没有逐字保留原文」的决定，写进决策日志。

4. **跑本块 `Verification` 的每一条。**任一不过，`git checkout -- .` 回滚本块的全部改动，转入报告，继续下一块。

   **规则块自己的 `Verification` 就是门。本手册不再设阶段级的门。**

   因为一个块只对自己的作用域负责。让它去过一道检查**全局**不变量的门，别的块的病会让它永远过不了 —— `legacy/doc-index-rot` 曾被 `CLAUDE.md` 的 transclusion 错误触发、又被同一个错误判死，而那个错误归 `doc-fork` 管、且排在它后面（证据：`docs/pitfalls.md` 第 11 条）。

5. **改动不得超出 `作用域`。**提交后 `git status --short` 必须为空 —— 非空即说明有作用域外的改动，那是越界：**中止整个阶段 3**，回滚未提交的改动，直接进入阶段 4 报告。

   这一条同时接住了「漏提交」和「越界」两种错误，不必分别检查。

6. **提交。**一个块一个 commit。`git add -A -- <作用域各项>`（必须覆盖作用域中的全部路径，含新建的未追踪文件）。commit message 写明规则块 id。任何一块改错都能单独 `git revert`。

退出条件：所有有 `作用域` 的块处理完毕。改动文件清单由 `git diff --name-only <起始commit> HEAD` 派生，不靠人工记账。

## 阶段 4：报告

向用户输出：

1. **决策日志 `.agents/refactor-decisions.md` 的全文。**这是每一个「没有逐字保留原文」的决定。**这份日志才是用户要读的东西，不是几百行的 diff。**
2. 阶段 1 的完整偏离清单，含被 `Do Not Apply When` 排除的项及其排除依据。
3. **没有 `作用域` 的块的命中**（`legacy/orphan-abstraction`、`legacy/vendored-knowledge`）—— 原样交出，每条附规则块 id 与证据位置。**不代为决策。**
4. 改动文件清单与 commit 列表。
5. 被中止的块及其原因。

## 双向沉淀

若某类腐烂在多个项目反复出现，走 `meta/rule-sedimentation` 的回流三判定。三条全「是」则抽象成新的 `legacy/` 规则块，补齐字段，写入框架 **git 工作副本**的 `reference/rules/legacy/`，并在回复中明确告知已回流的规则块 id。

不要把回流写进 `~/.claude/plugins/cache/` —— 因为该目录在插件更新时被整个替换，写进去的规则会无声消失。

回流后必须运行 `node scripts/validate-rules.mjs` 与 `node scripts/exercise-rules.mjs`，两者都返回 0 方可提交。

**后者会把你新写的 `Legacy Scan` 与 `Verification` 命令，在一个真实文件系统上执行一遍。**因为那些命令是用 markdown 写的小程序 —— 不跑它们，就只能等下一个用户在真实项目上第一次跑时，才发现它们是坏的（证据：`docs/pitfalls.md` 第 3、4、6、10、11 条，全部是规则块或手册里的命令在真实项目上才暴露的）。
